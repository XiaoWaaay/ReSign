# 项目文档（开发架构与实现说明）

## 1. 项目目标

ReSign 的目标是在“不改业务源码”的前提下，对目标 APK 做重打包注入，并在运行期提供一致性能力：

- 签名一致性：让目标应用通过 PackageManager/IPackageManager 读取到的签名与注入配置一致
- 路径/IO 视图一致：让目标应用看到的 base.apk 路径与反查路径更符合预期
- 入口交接稳定：替换入口后尽量把生命周期与组件创建交还给原始实现

---

## 2. 总体架构：pack-time + run-time

### 2.1 pack-time（重打包阶段）

核心入口：

- `app/src/main/java/com/xwaaa/resig/Core.java`

主要流程（逻辑上）：

1) 解析目标应用信息（base.apk 路径、split APK 列表）
2) 解包并提取：
   - dex
   - AndroidManifest.xml
3) 修改 Manifest 入口：
   - Application → `com.xwaaa.hook.HookApplication`
   - AppComponentFactory → `com.xwaaa.hook.HookApplication$DelegatingAppComponentFactory`
   - 写入 meta-data 配置（开关）
4) payload dex 注入：
   - 把 `assets/classesx.dex` 写入工作目录并重命名为 `classesN.dex`
   - 写入占位符（包名/签名/原 Application/原 Factory）
5) so / assets 注入：
   - 注入 `libkillsignture.so` 到 `lib/arm64-v8a/`
   - 注入 `assets/KillSig/origin.apk`（origin.apk 是 base.apk 的拷贝）
6) zip 增量更新（zip4j）

### 2.2 run-time（payload 运行期）

核心入口：

- 仓库根目录 `HookApplication.Java`（会被编译成 payload dex 并注入目标 APK）

运行期做两类事：

- Java Hook（Pine）
  - 在关键返回点统一 patch `PackageInfo`/`SigningInfo`
- Native redirect（seccomp + SIGSYS）
  - 对 open/readlinkat 等进行拦截与路径重定向，配合 origin.apk 使用
- Payload 文件落盘策略
  - `origin.apk` 使用“临时文件写入 + rename”方式落盘，避免半写入导致启动反复崩溃

运行期还负责交接：

- Application：attachBaseContext/onCreate 等转发给原 Application
- AppComponentFactory：尽量调用原 factory 创建四大组件

## 3. Pine 签名一致性：实现策略

目标：尽量减少 ROM/版本差异带来的维护成本，让“签名一致性”集中在少量 Hook 点上。

当前 Hook 点（run-time）：

1) `PackageInfo.CREATOR.createFromParcel(Parcel)`
2) `SigningInfo.CREATOR.createFromParcel(Parcel)`（Android 9+）
3) `ApplicationPackageManager` 的常用 API：
   - getPackageInfo / getPackageInfoAsUser / getPackageArchiveInfo
   - getInstalledPackages（列表场景）
4) `IPackageManager$Stub$Proxy`（若存在）
   - 扫描同名方法并 hook 返回 `PackageInfo` 或包列表的方法

Patch 方式：

- 若返回对象是目标包名：替换 `PackageInfo.signatures`，并递归 patch SigningInfo/SigningDetails 内的签名字段

对应实现位置：

- `HookApplication.Java` 的 `installPineSignatureHooksIfPossible(...)` 与相关 hook 方法

---

## 4. seccomp + SIGSYS：实现策略

目标：在 native 层做 syscall 拦截，让对 base.apk 的读取与 /proc/self/fd 反查结果更符合预期。

对应实现位置：

- `app/src/main/cpp/killsignture.cpp`

关键点：

- 使用 seccomp filter + SIGSYS handler 拦截目标 syscall
- 将 base.apk 访问重定向到 `/data/data/<pkg>/assets/KillSig/origin.apk`
- 对 readlinkat 结果进行伪造，避免直接暴露 origin.apk
- 对 `/proc/<pid>/maps` 的 openat 进行过滤（返回 memfd），降低 maps 扫描暴露风险

---

## 5. payload dex 的构建方式（工程关键点）

本项目 payload dex 不走 AGP 的“应用 classpath”，而是独立脚本生成：

- `build.sh`：用 `javac + d8` 生成 payload dex
- `:app:preparePayloadDeps`：抽取 Pine AAR 的 `classes.jar` 到 `app/build/payload-deps`

因此：

- 仅在 `app/build.gradle.kts` 里加 implementation 不够
- 必须确保 Pine 的 classes 被 d8 输入合并进 `classesx.dex`

pack-time 对 payload dex 的“占位符写入”实现：

- `Inject.kt` 的 `editShellDEX(...)` 使用 dexlib2 重写 dex 的字符串常量
- 不再依赖 baksmali/smali 反编译/回编译流程（减少耗时与不确定性）

## 6. 目录与模块速览

- `HookApplication.Java`：payload 入口（Java hook + native redirect + 交接）
- `app/src/main/java/com/xwaaa/resig/Core.java`：重打包主流程
- `app/src/main/java/com/xwaaa/resig/Inject.kt`：Manifest 修改与 payload 占位符写入
- `app/src/main/java/com/xwaaa/resig/FileUtils.kt`：文件/zip 工具
- `app/src/main/cpp/killsignture.cpp`：seccomp + SIGSYS native 实现
- `docs/使用文档.md`：使用侧说明
