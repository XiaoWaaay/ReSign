# ReSign 稳定性优化方案（优先落地）

本文档聚焦“稳定性优先”的优化方向：在不改变 ReSign 现有产品定位（pack-time 重打包 + run-time 一致性修复）的前提下，最大化降低崩溃率、兼容性风险与不可观测问题带来的维护成本，并给出可分阶段实施的落地清单。

## 0. 目标与原则

### 0.1 总体目标（可度量）

- 启用默认配置时：目标应用启动成功率显著提升，且出现异常可自动降级恢复可用
- 在 ROM/版本差异较大的场景：run-time 能力按风险分级启用，避免“一启用就崩”
- pack-time 重打包结果具备可验证性：失败能明确定位阶段，避免半成品产物

建议核心指标（可按实际补充采集方式）：

- 启动成功率：安装后首次启动/冷启动成功率
- 崩溃率：进程启动 60 秒内崩溃次数
- 降级有效率：触发降级后能否恢复到可启动状态
- 能力命中率：签名 patch 命中次数、native 重定向后端是否安装成功

### 0.2 设计原则（稳定性优先）

- 默认保守：SAFE/标准模式尽量不使用高风险能力
- 失败可预期：关键路径不“吞异常”，失败要可见、可复现、可回退
- 分阶段启用：run-time 初始化拆阶段，失败立即停止并落盘记录，避免链式故障
- 单向兼容：配置与协议可版本化，旧产物在新 payload 上尽量可运行（或能安全降级）

## 1. 当前薄弱点与风险分级

### 1.1 pack-time（重打包侧）主要风险

- ZIP 增量写入次数多：多 dex 场景下反复 remove+add，耗时高且失败后难定位
- 缓存键偏重哈希：对大 APK 的 sha256 扫描可能引入明显耗时
- 失败处理偏“best-effort”：部分异常捕获后继续，可能留下半成品 base.apk
- 输出链路不闭环：产物签名/安装流程不在项目内，用户失败后排查成本高

### 1.2 run-time（payload 侧）主要风险

- Native 拦截（seccomp + SIGSYS）对 ROM/内核差异敏感：失败形态复杂（拒绝安装 filter、SIGSYS 行为差异、线程同步失败等）
- Hook 点覆盖面广但观测弱：Java hook 安装/命中情况缺少统一、结构化的诊断输出
- “隐藏”类能力合规与稳定性风险较高：maps hide、deep hide 等可能导致不可预期问题

### 1.3 风险分级（建议）

- 低风险（默认开启）：签名一致性（Java hook）中的“返回值 patch”类逻辑
- 中风险（按模式开启）：IO/路径重定向（native），需严格分阶段探测与降级
- 高风险（默认关闭，仅研究/测试）：maps hide、deep hide、seccomp full aggressive

## 2. 优化路线图（按优先级）

建议以“稳定性第一、性能第二、隐蔽第三”的顺序推进。

### 阶段 A（优先落地）：稳定性与可观测性补齐

目标：即使出现问题，也能快速定位与自动降级恢复。

1) run-time：Native 安装改为分阶段探测与落盘记录（强烈推荐）
2) run-time：统一诊断输出（仅 debugLog 开启），包含关键状态与错误码
3) pack-time：重打包流程阶段化，失败明确归因，避免半成品产物导出

### 阶段 B：兼容性增强（减少“某机型不可用”）

1) run-time：引入“进程维度开关”（主进程优先，非主进程默认关闭高风险能力）
2) run-time：将 hookMode 与 native backend 做明确映射，并支持强制回退 SAFE
3) pack-time：split 场景交付形态增加校验与提示（先可诊断，后可一键安装）

### 阶段 C：性能优化（不牺牲稳定性）

1) pack-time：批量 zip 更新，减少反复打开 zip 的次数
2) pack-time：缓存键优化，避免对大 APK 频繁全量哈希

## 3. 具体改进项（目标 / 步骤 / 预期效果）

### 3.1 run-time：Native 安装分阶段探测（稳定性核心）

改进目标：

- native 能力的失败不会导致链式崩溃
- 能明确知道失败发生在“加载 so / SIGSYS / seccomp filter / 重定向规则”哪个阶段

实施步骤（建议实现形态）：

- 将 native 初始化拆分为四阶段，每阶段单独 try/catch，并产出结构化状态
  - Phase 1：加载 so（System.loadLibrary）
  - Phase 2：启用 debug 日志开关（可选）
  - Phase 3：准备 origin.apk（提取并原子落盘）
  - Phase 4：安装重定向（调用 hookApkPathWithBackend），并读取 getRedirectBackend 校验
- 每阶段失败：
  - 立即停止后续阶段
  - 记录错误码 + 栈摘要（仅 debugLog）
  - 更新“降级状态”（见 3.2）

预期效果：

- 大幅降低“native 一启用就崩”带来的不可恢复问题
- 真机问题定位效率提升（阶段 + 错误码明确）

### 3.2 run-time：降级策略标准化（崩溃可恢复）

改进目标：

- 当连续启动崩溃发生时，自动降级到 SAFE（或禁用 native），保证可用性

实施步骤：

- 现有 crash state 机制保留，但增强为“能力级别降级”
  - Level 0：仅 Java hook
  - Level 1：Java hook + 低风险 native（非 full）
  - Level 2：开启 aggressive（不建议默认）
- 在降级触发条件上引入“时间窗 + 次数阈值”
- 降级落盘写入必须原子化（临时文件 + rename）

预期效果：

- “坏配置/坏机型”从不可用变为可自动恢复

### 3.3 run-time：诊断面（仅 debugLog）统一化

改进目标：

- 能回答“为什么没生效/为什么崩溃”：hook 是否安装成功、命中多少次、native 后端是什么

实施步骤：

- 输出统一的诊断块（建议单行 key=value，便于 logcat 过滤）
- 关键字段建议包含：
  - payloadId
  - targetPkg
  - hookMode
  - enableJavaHook / enableNativeHook / enableIoRedirect / enableMapsHide / enableDeepHide
  - Java hook：已 hook 方法数、patchPackageInfo 命中次数
  - native：backend（getRedirectBackend）、origin.apk 路径与大小、安装阶段结果

预期效果：

- 线上/真机回归时快速定位“不生效”的根因

### 3.4 pack-time：流程阶段化与结果校验（避免半成品）

改进目标：

- pack-time 任意阶段失败都能输出“失败阶段 + 失败原因”
- 不导出可能无法安装/无法启动的半成品 APK

实施步骤：

- 将重打包拆为明确阶段：
  - copy base.apk → extract dex/manifest → patch payload dex → patch manifest → inject so/assets → write back zip → export
- 在 write back zip 后做最小校验：
  - Manifest 入口是否已替换
  - payload dex 是否存在且为 classesN.dex
  - assets/KillSig/origin.apk 是否存在
  - lib/<abi>/libkillsignture.so 是否存在（或按选项关闭）
- 校验失败则不导出，并给出明确提示

预期效果：

- 明显减少“看似成功但启动必崩”的产物
- 缩短排查时间

### 3.5 pack-time：批量 zip 更新（性能提升但不牺牲稳定性）

改进目标：

- 在多 dex 的场景显著降低耗时与失败概率

实施步骤（建议先做易落地版本）：

- 优先把 dex 注入从“每个 dex 一次 addToZip”改为“批量 add”
- 若仍使用 zip4j：
  - 复用 ZipFile 实例，减少反复打开
  - 对同一 entry 的 remove+add 合并为一次操作

预期效果：

- 重打包时间下降，且稳定性更好（更少 I/O 操作、失败点更少）

## 4. 开关策略建议（默认配置）

建议将默认 UI 配置改为保守（以稳定为第一目标）：

- hookMode：默认 STANDARD（或 SAFE，取决于目标应用覆盖面）
- enableMapsHide：默认关闭，仅 AGGRESSIVE + 手动开启
- enableDeepHide：默认关闭
- enableNativeHook / enableIoRedirect：默认开启但需分阶段探测，失败自动回退
- debugLog：默认关闭；问题定位时开启

## 5. 验证与回归建议（稳定性工程化）

### 5.1 最小回归矩阵（建议）

- Android 版本：10/12/13/14（至少 12/13/14）
- ABI：arm64-v8a（当前项目默认）
- ROM：AOSP + 1~2 个主流厂商 ROM

### 5.2 回归用例（建议自动化优先）

- pack-time：
  - 重打包成功率（含多 dex、split）
  - 产物结构校验（Manifest/DEX/assets/so）
- run-time：
  - 启动不崩溃（冷启动 3 次）
  - Java 签名查询一致性（PackageManager/Parcel 兜底）
  - native 重定向测试（已有 NativeRedirector instrumentation 可复用）
  - 降级机制：模拟连续崩溃后能进入 SAFE 并可启动

## 6. 里程碑与交付物

### M1（优先）

- native 分阶段探测 + 结构化诊断输出
- pack-time 阶段化结果与最小校验

### M2

- 进程维度开关与更明确的 mode/backend 映射
- split 交付提示与校验增强

### M3

- 批量 zip 更新与缓存键优化
